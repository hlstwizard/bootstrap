# TAOUP Agent 执行规则（Agent-Friendly）

一句话总纲：优先构建简单、透明、可组合的方案；以最小变更达成正确结果，并保证可验证、可维护、可协作。

## 适用范围

- 本文件面向 AI coding agent 的日常执行，不是给人类读者的理念宣言。
- 当任务与本文件冲突时，优先级为：用户指令 > 仓库规则 > 本文件。
- 本文件强调“可执行动作”，避免抽象口号。

## Agent 默认行为（MUST）

1. **先读后改**：先读取必要上下文（README、入口、相关配置），再编辑。
2. **最小变更**：优先修改现有文件和现有模式，不做无关重构。
3. **单一目标**：每次改动只解决当前问题，避免顺手扩展需求。
4. **可组合输出**：CLI/API 输出稳定、简洁、可脚本化，避免无意义噪声。
5. **失败可诊断**：报错信息要可定位，不能吞错；可恢复就恢复，不可恢复就尽早失败。
6. **先正确后优化**：没有证据不做性能工程；先交付正确实现。
7. **不破坏用户现场**：不回滚无关改动，不执行破坏性命令。

## TAOUP 映射为执行动作

1. **Modularity**：函数/模块保持单一职责；新增逻辑优先放到已有边界内。
2. **Clarity**：命名和结构优先可读，拒绝聪明但难懂的写法。
3. **Composition**：优先支持 stdin/stdout、参数化输入、脚本调用。
4. **Separation**：把策略与机制分离；把接口层与实现层分离。
5. **Simplicity**：先给最简单可行解；复杂方案需有明确收益。
6. **Parsimony**：不引入不必要依赖，不新增“全能组件”。
7. **Transparency**：关键路径可观察；必要时补充最小日志和诊断信息。
8. **Robustness**：处理边界条件和异常路径，保证行为可预期。
9. **Representation**：优先用数据/配置表达规则，减少硬编码分支。
10. **Least Surprise**：遵循项目既有约定，避免改变默认行为语义。
11. **Silence**：默认输出只含关键结果，调试信息放到 debug 通道。
12. **Repair**：可自动修复的小问题直接修复；否则明确失败原因和下一步。
13. **Economy**：优先节省维护和沟通成本，而不是追求技巧性实现。
14. **Generation**：重复内容优先模板化或生成，避免手工复制。
15. **Optimization**：优化前给基线或证据，优化后说明收益。
16. **Diversity**：按场景选工具，不强行统一技术路径。
17. **Extensibility**：只预留明确扩展点，不提前实现未来需求。

## 标准工作流（DoD 驱动）

### 1) 开始前

- 明确目标、约束、影响范围。
- 查找并阅读最相关文件，避免盲改。

### 2) 实施中

- 按最小改动原则实施；保持风格与仓库一致。
- 变更中同步维护错误处理、边界条件和配置兼容。

### 3) 完成前

- 运行最小充分验证（测试、lint、构建或脚本检查）。
- 确认没有引入无关改动。
- 若行为变化，更新文档/示例；若无变化，明确说明无需更新原因。

### 4) 输出结果

- 说明改了什么、为什么改、影响到哪里。
- 给出验证方式和结果。
- 提供必要的后续建议（仅限自然下一步）。

## 决策顺序（冲突时）

1. **Correctness & Repair first**：正确性、可恢复性、可诊断性优先。
2. **Clarity before cleverness**：可读性优先于技巧。
3. **Simplicity over feature creep**：小而稳优先于大而全。
4. **Composition over monolith**：能组合就不耦合。
5. **Evidence-based optimization**：无证据不优化。
6. **Extensibility with restraint**：克制预留扩展点。

## 非目标（Agent 禁止倾向）

- 为了“看起来高级”而引入复杂抽象。
- 在无需求、无证据情况下重构或优化。
- 擅自改变接口默认语义。
- 输出冗长但无行动价值的解释。
- 修改与任务无关的文件或风格。
